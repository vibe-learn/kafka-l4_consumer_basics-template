        # kafka — Консьюмер, группа и оффсеты

        Homework-шаблон для урока **l4_consumer_basics** (Консьюмер, группа и оффсеты) на платформе Vibe Learn.

        ## Что делать

        Реализуй consumer group с manual commit, демонстрирующую at-least-once семантику.
Два инстанса консьюмера читают топик `task_events` (8 партиций). Продьюсер пишет 100
сообщений. Обработка каждого сообщения намеренно медленная (sleep 100 мс). В середине
прогона один инстанс убивается через `kill -9`. После перезапуска CI проверяет:
итоговое число успешно обработанных сообщений >= 100 (переобработка допустима, потеря — нет).

## Контекст (из transfer-задачи урока)

В вашем сервисе уведомлений 4 инстанса консьюмера читают топик `user_notifications` с
8 партициями (consumer group `notification-sender`). Каждое сообщение отправляется в
push-сервис через HTTP, который иногда тормозит (p99 = 2 с). Сейчас конфигурация:
`enable.auto.commit=true`, `auto.commit.interval.ms=5000`.

Бизнес жалуется на два симптома одновременно:
- Иногда пользователи **не получают** пуш-уведомление вовсе.
- Иногда пользователи **получают то же уведомление дважды**.

## Recap из урока

- **Consumer group** — единица потребления: Kafka назначает каждой партиции ровно одного консьюмера в группе. Лишние консьюмеры (больше числа партиций) будут idle.
- **Оффсеты** хранятся в системном топике `__consumer_offsets`. Момент коммита определяет гарантию доставки — не Kafka, а ваш код.
- **Auto-commit** удобен, но опасен при долгой обработке: коммит происходит по таймеру, не по факту завершения бизнес-логики → возможна потеря сообщений при падении (at-most-once).
- **Manual commit after processing** — at-least-once: сообщение может быть переобработано, но никогда не потеряно. Для большинства задач это правильный выбор.
- **Rebalance** происходит при изменении набора консьюмеров или числа партиций. Во время rebalance группа не читает — это «остановка мира» для консьюмеров.

        ## Как работать

        1. Платформа Vibe Learn создаёт копию этого репо в твоём GitHub-аккаунте по клику «Начать домашку» на странице урока (через GitHub `/generate`, codecrafters-pattern).
        2. Склонируй копию локально, реализуй TODO в `main.go`, прогони тесты, запушь.
        3. CI (`.github/workflows/ci.yml`) запускает `go vet` + `go test ./...` на каждый push. Платформа слушает результат через webhook от GitHub Actions и обновляет статус домашки на странице урока.

        ## Локальное окружение

        - Go 1.22+
        - Docker + docker-compose — `docker compose -f docker-compose.yml up -d` поднимает 3-нодовый Kafka cluster на портах 9092/9093/9094, использовать в тестах через bootstrap `localhost:9092,localhost:9093,localhost:9094`.

        ## Запуск

        ```bash
        # Поднять локальный Kafka
        docker compose up -d

        # Прогнать тесты (часть из них стартует свой ephemeral testcontainers cluster, часть использует docker-compose выше)
        go test ./...

        # Запустить main (печатает marker; замени stub на реализацию)
        go run .
        ```
